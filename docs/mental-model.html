The Mental Model

Native Federation is built on the proven mental model of Module Federation — a set of concepts that enable runtime integration of separately built and deployed applications.

Runtime Integration

The core idea is simple: load a part of a separately built and deployed application into yours at runtime. This is the foundation for both Micro Frontend architectures and plugin-based solutions.

Unlike traditional lazy loading where everything is known at build time, federation allows loading code that the host application has never seen during compilation. This enables truly independent development and deployment cycles across teams.

Key Concepts

Remote

A remote is a separately built and deployed application that exposes EcmaScript modules for other applications to consume. A remote can expose anything — a single component, a full routing configuration, or any other TypeScript/JavaScript construct.

Each remote publishes a remoteEntry.json metadata file that describes what it exposes and which shared dependencies it uses.

Host

The host (or shell) loads one or more remotes on demand. From your framework's perspective, this looks exactly like traditional lazy loading — the crucial difference is that the host doesn't know about the remotes at compile time.

A host reads a federation manifest at runtime that maps remote names to their entry point URLs. This manifest can be swapped during deployment, allowing you to adapt the build to different environments.

Shared Dependencies

When multiple remotes and the host use the same library (e.g., Angular, RxJS), you don't want to download it multiple times. Shared dependencies are downloaded once and reused at runtime across all federated applications.

Native Federation places shared libraries into their own standards-compliant EcmaScript bundles and uses Import Maps to tell the browser where to load each one from.

Version Mismatch Handling

When two or more applications require different versions of the same shared library, Native Federation provides several strategies:

  • Compatible fallback: Use a different version that satisfies semantic versioning constraints.
  • Strict mode: Throw an error if no compatible version is found, preventing subtle runtime bugs.
  • Singleton enforcement: Guarantee that only one instance of a library exists, crucial for stateful libraries like Angular.

How It Works Under the Hood

At runtime, Native Federation uses Import Maps — a web standard for mapping module specifiers to URLs. During the build, Native Federation:

  1. Analyzes shared dependencies and creates individual ESM bundles for each.
  2. Generates metadata files (remoteEntry.json) describing exposed modules and required shared dependencies.
  3. At runtime, reads the federation manifest and remote metadata to construct an Import Map.
  4. The browser uses this Import Map to resolve and load modules on demand.

Because Import Maps are a browser-native standard, Native Federation is not tied to any specific build tool. It works with esbuild, webpack, or any future bundler.
