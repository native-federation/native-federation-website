Orchestrator

Applies to

Fully compatible with v3 and v4.
Although the orchestrator was introduced in native federation v4, version 3 and 4 contain the same runtime spec (remoteEntry.json), so the orchestrator makes can seamlessly load v3 remotes as well as v4 remotes.

The Orchestrator — @softarc/native-federation-orchestrator — is the next-generation browser runtime for Native Federation. It is intended to replace the default Runtime (@softarc/native-federation-runtime) as the recommended way to load remotes on the host, regardless of whether the host is a SPA, a plain HTML page, or a server-rendered application.

Compared to the default runtime it adds semver-range resolution when remotes disagree on a shared dependency version, persistent caching of remoteEntry.json in localStorage or sessionStorage, and a drop-in quickstart bundle for HTML-only hosts. It stays fully compatible with the Native Federation ecosystem — any remote built with @softarc/native-federation (v3 or v4) that emits a standard remoteEntry.json can be loaded by it.

The one caveat today is SSR: the Orchestrator runs in the browser and does not yet have direct server-side rendering support. A server-rendered host works fine — the Orchestrator loads the remotes client-side after the page arrives — but if you need the remote modules to execute during SSR itself, stick with the default Runtime for now.

The Orchestrator is the new recommended runtime introduced in v4. For a full comparison with the v3 Classic Runtime — when to use which, semver resolution, and caching — see v3 vs v4.
