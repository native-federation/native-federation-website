Architecture Overview

Native Federation is organized into four cooperating layers. Each has a single responsibility and a narrow contract with its neighbors — so you can replace any one of them (your framework, your bundler, or your runtime) without touching the others.

The Four Layers

LayerLives atRunsResponsibility
Core @softarc/native-federation build time Normalizes the federation config, bundles shared dependencies and exposed modules, and emits remoteEntry.json + the import map. Bundler-agnostic.
Adapters Angular / esbuild / Vite … build time Plug a specific bundler or framework into the Core via the NFBuildAdapter contract. Often ship a higher-level API, schematics or CLI integration on top.
Runtime in the browser run time Reads remoteEntry.json files, constructs a combined import map, and loads remote modules on demand. Small, framework-agnostic.
Orchestrator in the browser run time The next-generation runtime, intended to replace the default Runtime. Same remoteEntry.json contract, plus semver-range resolution for shared dependencies and persistent caching in browser storage. Client-side only — no direct SSR support yet.

Core

The Core — @softarc/native-federation — is the heart of the framework. It is deliberately framework- and bundler-agnostic: it only knows about shared dependencies, exposed modules, import maps and a pluggable build adapter.

Given a federation.config.js and an adapter, the Core:

  1. Normalizes your config (defaults, skip list, mapped paths, version inference).
  2. Computes the list of externals — the packages your own bundler must leave unresolved.
  3. Bundles every shared external, every shared tsconfig path, and every exposed module.
  4. Writes remoteEntry.json and importmap.json.
  5. Caches work keyed by content checksum so rebuilds stay fast.

Adapters

Adapters are the bridge between the Core and a specific build tool or framework. Each adapter is an object implementing NFBuildAdapter — three methods (setup, build, dispose) that the Core calls to bundle a group of entry points.

Most adapters also layer framework-specific ergonomics on top: the Angular adapter adds an Angular CLI builder and schematics; the esbuild adapter adds React-friendly fileReplacements for CommonJS interop.

  • Angular — first-class Angular CLI integration.
  • esbuild — framework-agnostic; powers the React reference example.
  • Vite — community plugin.
  • Your own — any bundler that can accept an external list and emit ESM can be wrapped in an adapter.

→ Adapters docs

Runtime

The Runtime is the piece that ships to the browser. It is tiny by design: it fetches the remoteEntry.json of each remote listed in the host manifest, merges their shared-dependency declarations, and constructs a single import map for the page. From that point on the browser itself resolves every import — including those inside lazily-loaded remotes.

The Runtime exposes two functions:

  • initFederation(manifestOrMap) — called once at startup in both hosts and remotes.
  • loadRemoteModule({ remoteName, exposedModule }) — called whenever a host wants to pull in an exposed module.

→ Runtime docs

Orchestrator

The Orchestrator — @softarc/native-federation-orchestrator — is the next-generation browser runtime, intended to replace the default Runtime as the recommended way to load remotes on the host. It speaks the same remoteEntry.json contract and exposes the same initFederation / loadRemoteModule surface, so remotes built by the Core work unchanged. On top of that it adds semver-range resolution when remotes disagree on a shared version, and persistent caching of remoteEntry.json in localStorage or sessionStorage.

It works in SPAs, plain HTML pages, and server-rendered hosts alike. The one limitation today is SSR: the Orchestrator runs client-side only and does not yet have direct server-side rendering support — if you need the remote modules to execute during SSR itself, stick with the default Runtime.

→ Orchestrator docs

A Full Build-to-Runtime Trace

  1. Your build pipeline invokes the Native Federation build for shell and mfe1, each with their own FederationOptions.
  2. For each project, the Core reads federation.config.js, normalizes it, and asks the Adapter to bundle shared externals, mapped paths, and exposed modules.
  3. The Core writes dist/<project>/remoteEntry.json and dist/<project>/importmap.json.
  4. The dist folders are published behind stable URLs.
  5. At startup, the Runtime (or the Orchestrator) in shell fetches mfe1's remoteEntry.json, merges shared-dependency metadata, resolves version mismatches, and injects a combined import map into the document.
  6. When the user navigates to a route backed by a remote, loadRemoteModule dynamically imports the exposed module — the browser resolves it through the injected import map.

If you only remember one thing: the Core and Runtime speak a simple contract — remoteEntry.json plus an import map. Everything else (which bundler, which framework, which runtime) is swappable.
